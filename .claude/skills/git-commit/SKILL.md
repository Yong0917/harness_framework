변경된 파일들을 분석하여 기능/목적별로 그룹화하고, 각 그룹에 맞는 의미 있는 커밋을 생성하라.

---

## 워크플로우

### 1. 변경사항 파악

아래 명령어로 현재 변경된 파일 전체를 확인한다:

```bash
git status
git diff --stat HEAD
git diff --cached --stat
```

staged와 unstaged 파일을 모두 포함한다.

### 2. 파일 분석

변경된 파일 각각에 대해 다음을 파악한다:

- **목적**: 이 파일이 무엇을 하는가?
- **변경 이유**: 왜 이 파일이 변경되었는가?
- **연관성**: 다른 어떤 파일과 함께 변경되었는가?

`git diff <file>` 또는 파일 읽기로 실제 변경 내용을 확인한다. 파일 이름만 보고 추측하지 마라.

### 3. 그룹화

연관된 파일들을 의미 있는 단위로 묶는다. 그룹화 기준:

- **기능 단위**: 하나의 기능을 구현하는 파일들 (예: 소스 코드 + 그에 대응하는 테스트)
- **목적 단위**: 같은 목적을 가진 변경들 (예: 설정 파일들, 문서들)
- **레이어 단위**: 같은 레이어에 속하는 변경들 (예: API 레이어, DB 레이어)

그룹화 판단이 애매한 경우:
- 테스트 파일은 해당 소스 파일과 같은 그룹에 넣는다
- 설정/환경 파일은 별도 그룹으로 분리한다
- 문서는 별도 그룹으로 분리한다
- 서로 독립적인 버그 픽스는 각각 별도 그룹으로 분리한다

### 4. 커밋 메시지 작성

각 그룹에 대해 conventional commits 형식으로 커밋 메시지를 작성한다:

```
<type>(<scope>): <subject>

[optional body]
```

**type 선택 기준:**
- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `refactor`: 기능 변경 없는 코드 개선
- `test`: 테스트 추가/수정
- `docs`: 문서 변경
- `chore`: 빌드 스크립트, 설정 파일, 의존성 변경
- `perf`: 성능 개선
- `style`: 포맷팅, 세미콜론 등 코드 의미 변경 없는 수정

**subject 작성 원칙:**
- 50자 이내, 현재형 동사로 시작 (한국어는 동사 원형)
- 무엇을 했는지가 아니라 **왜/무엇을** 했는지를 담는다
- "파일 수정" 같은 무의미한 메시지 금지

### 5. 커밋 실행

그룹별로 순서대로 커밋한다:

1. 각 그룹의 파일을 staging한다: `git add <files>`
2. 커밋을 생성한다 (Co-Authored-By 없이):

```bash
git commit -m "$(cat <<'EOF'
<type>(<scope>): <subject>
EOF
)"
```

여러 줄이 필요한 경우:
```bash
git commit -m "$(cat <<'EOF'
<type>(<scope>): <subject>

<body>
EOF
)"
```

**절대 `Co-Authored-By:` 줄을 추가하지 마라.**

### 6. 결과 보고

커밋 완료 후 아래 형식으로 요약한다:

```
커밋 완료 (N개):

1. <hash> <type>(<scope>): <subject>
   파일: <file1>, <file2>

2. <hash> <type>(<scope>): <subject>
   파일: <file3>
```

`git log --oneline -N` 으로 최신 N개 커밋을 확인한다.

---

## 주의사항

- 파일 내용을 실제로 읽어 변경 의도를 파악한 뒤 그룹화한다. 파일 이름만으로 판단하지 마라.
- 하나의 커밋에 너무 많은 파일을 몰아넣지 마라. 리뷰하기 어려워진다.
- 반대로 같은 기능을 구현하는 파일들을 불필요하게 쪼개지 마라.
- `git push` 는 사용자가 명시적으로 요청할 때만 실행한다.
- 커밋 메시지에 `Co-Authored-By:` 를 절대 추가하지 마라.
